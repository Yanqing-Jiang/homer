## Key Design Decisions

### 1. Proposals are Immutable-ish
When a proposal advances stage (idea â†’ plan), we can either:
- **Option A**: Update in place (current design) - simpler
- **Option B**: Create new row, link via `parent_proposal_id` - full audit trail

Current design uses Option A with `approval_history` for audit.

### 2. Runs vs Intents
- **Intent**: What we want to do (can have multiple attempts)
- **Run**: A single execution attempt

An Intent with `status=failed` can be retried, creating a new Run with incremented `attempt_number`.

### 3. Executor Account Rotation
Multiple accounts per executor type enables:
- Rate limit avoidance (rotate when one hits limits)
- Cost distribution (spread across accounts)
- Failure isolation (disable problematic accounts)

### 4. Approval Learning
`approval_history` captures every decision with context, enabling future:
- Auto-approval rules based on patterns
- Source reputation scoring
- Risk calibration

### 5. Backward Compatibility
Migration preserves:
- `pending_plans` â†’ `proposals` (stage: plan)
- `idea_index` â†’ `proposals` (stage based on old status)
- Existing `job_queue` for Telegram-driven jobs (separate from runtime intents)
